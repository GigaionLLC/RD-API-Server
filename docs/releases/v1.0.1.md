# RD-API-Server v1.0.1

**Release date:** 2026-07-17

**Status:** Stable patch release

RD-API-Server v1.0.1 makes dark mode the predictable first-visit experience and removes the
emulated ARM build bottleneck from container publishing. It is a backward-compatible patch over
v1.0.0: there are no database migrations, RustDesk client wire changes, route changes, or new
production configuration requirements.

> RD-API-Server is an independent, third-party open-source project. It is not affiliated with,
> endorsed by, or sponsored by RustDesk or Purslane Ltd. RustDesk is referenced only to describe
> compatibility with the open-source client's public API.

## What changed

### Dark mode by default

- Authentication, two-factor challenge, and administration pages now start in dark mode when the
  browser has no saved RD-API-Server theme preference.
- A user-selected light or dark preference remains persistent across navigation, reloads, and
  later operating-system theme changes.
- The operating system no longer silently changes an explicit application theme. Select the
  theme toggle in the administration header to switch modes.
- Light-mode accessibility and interaction coverage remains in the browser test matrix through an
  explicitly saved light preference.

### Faster, safer image builds

- The production Dockerfile compiles its native PHP extension set once per architecture. Composer
  dependency assembly and the final Apache runtime reuse that exact digest-pinned PHP layer.
- Locked Composer packages are cached before application source is copied, so ordinary PHP, Blade,
  JavaScript, and documentation edits do not force another dependency download.
- AMD64 and ARM64 images build concurrently on native GitHub-hosted runners. The release workflow
  no longer uses QEMU to compile ARM dependencies.
- Each architecture is pushed by content digest, pulled back onto matching hardware, and checked
  for PHP 8.5, required extensions, the optimized autoloader, Apache configuration and rewrite
  support, and exclusion of build-only Composer/extension tools.
- Final `latest`, SemVer, and commit tags move only after the exact source commit passes PHP,
  JavaScript, vendor-integrity, Playwright, both architecture smoke tests, and digest validation.
  The publication job succeeds only after it then verifies the final manifest, provenance, and
  every generated tag.
- Architecture-specific GitHub caches plus main-owned registry cache fallbacks speed repeat builds
  without changing the digest-pinned base images, locked dependencies, or final image contents.

## Install or pin v1.0.1

Use the versioned release tag when a deployment must remain on this patch:

```yaml
services:
  rustdesk-api:
    image: ghcr.io/gigaionllc/rustdesk-api-server:1.0.1
```

The release also publishes compatible `1.0` and `1` tags. For the strongest deployment
reproducibility, pin the selected version tag together with its published multi-architecture
manifest digest.

Follow the [Quick Start](../../QUICKSTART.md) for the complete Docker and MariaDB configuration.
Production deployments require unique administrator and database passwords; do not copy example
placeholders into a live environment.

## Upgrade from v1.0.0

1. Back up MariaDB, the persistent application storage, and the matching application key.
2. Pull the `1.0.1` image and recreate the application container.
3. Confirm `/api/version` reports `1.0.1` and open `/admin` in a private browser session to verify
   the new first-visit dark default.
4. If the application is behind a TLS reverse proxy, retain the exact application-observed
   `TRUSTED_PROXIES` address or narrow isolated-network CIDR, `APP_URL=https://...`, and
   `SESSION_SECURE_COOKIE=true`; then run the public HTTPS checker.

No schema migration is introduced by this patch. The v1.0.0 boundaries remain unchanged:
MariaDB/InnoDB is the only supported database, `DB_URL` is rejected, wildcard proxy trust is not
supported, and a proxy-bypass path must not expose the application backend.

```bash
docker compose pull rustdesk-api
docker compose up -d --force-recreate rustdesk-api
scripts/check-https-proxy.sh https://api.example.com
```

Direct local HTTP deployments should continue to leave proxy trust empty and use request-based
cookie detection instead of forcing secure cookies.

## Verification

The release candidate passed the repository's Docker-based verification matrix, including:

- 538 PHPUnit tests with 3,051 assertions against an isolated MariaDB/InnoDB database, Pint across
  275 files, and PHPStan across 177 files.
- ESLint and checked-in frontend vendor-integrity validation.
- 69 passing Playwright tests with 15 intentional screenshot/project skips across desktop
  dark/light, tablet dark, and mobile dark, including the new OS-light first-visit and
  persisted-theme cases.
- Strict Composer validation, Compose rendering, Bash and GitHub workflow lint, immutable action
  pin checks, and local documentation-link validation.
- A fresh PHP 8.5 runtime build, production dependency platform validation, final-image exclusion
  of build-only tools, and successful startup/migration against disposable MariaDB/InnoDB with
  `/up`, the themed admin login, and `/api/version` checks.
- The release CI graph additionally requires native AMD64/ARM64 digest builds, per-architecture
  smoke checks, and final manifest/provenance verification before versioned images can publish.
