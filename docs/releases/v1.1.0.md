# RD-API-Server v1.1.0

**Release date:** 2026-07-18

**Status:** Stable minor release

RD-API-Server v1.1.0 replaces the production Apache runtime with a supervised Nginx/PHP-FPM
container and adds an explicit wildcard reverse-proxy trust mode for simple isolated deployments.
It is a backward-compatible application upgrade over v1.0.1: the container still serves port
`80`, persists `/var/www/html/storage`, uses MariaDB/InnoDB, and preserves every RustDesk client
route, JSON key, and response shape. There is no database migration in this release.

> RD-API-Server is an independent, third-party, open-source project. It is not affiliated with,
> endorsed by, or sponsored by RustDesk or Purslane Ltd. RustDesk is referenced only to describe
> compatibility with the open-source client's public API.

## What changed

### Nginx and PHP-FPM production runtime

- Nginx serves static assets and forwards only Laravel's exact front controller to PHP-FPM over a
  permission-controlled Unix socket. PHP-FPM does not expose or listen on TCP port `9000`.
- A `tini`-backed supervisor treats an unexpected Nginx or PHP-FPM exit as a container failure and
  performs a bounded graceful drain for both `SIGQUIT` and `SIGTERM`.
- The bundled Compose files retain container port `80`, the persistent storage path, application
  environment variables, MariaDB startup/migrations, and the one-service upgrade workflow.
- Request-body limits follow the configured recording chunk with headroom. Nginx owns the single
  optional access log, while PHP-FPM worker and slow-request settings are range-checked before
  migrations or first-run seeding can change persistent state.
- Native AMD64 and ARM64 release jobs start the real image with disposable MariaDB and verify
  Nginx/FPM syntax, socket isolation, HTTP/API behavior, proxy recovery, body/path boundaries,
  process failure, graceful shutdown, and removal of build-only tooling.

### Reverse-proxy convenience and restrictions

- `TRUSTED_PROXIES=*` is now an explicit supported value. Bundled Compose files use `*` when the
  variable is unset so TLS-terminating reverse proxies work without discovering a dynamic Docker
  peer address first.
- Wildcard mode trusts `X-Forwarded-For` and `X-Forwarded-Proto` from every immediate caller and
  emits a startup warning. Forwarded host, port, and prefix remain ignored.
- Wildcard mode is not recommended when any caller can reach the application HTTP port without
  passing through the sanitizing proxy. Such a caller could spoof the address used by login
  throttles, API-key IP allowlists, and audit records.
- The recommended production setting remains the proxy's exact application-observed IP address
  or a narrow isolated proxy-network CIDR. Multiple restricted entries are comma-separated. An
  explicitly empty value disables forwarded-header trust for direct HTTP operation.
- `**`, `REMOTE_ADDR`, IPv4/IPv6 `/0`, malformed entries, and implicit wildcard equivalents remain
  rejected. If `*` appears with narrower entries, wildcard mode wins; use it alone.

## Capacity scope

The release passed the full functional and native-container matrix plus a short source-matched
Apache/Nginx tuning comparison at 300 heartbeat requests per second. Both runtimes completed the
schedule without failures, drops, or wire mismatches at roughly 7 ms p95; the Nginx container used
less sampled application memory.

That tuning run is not certification for a 10,000-device fleet. The planned 1,980-RPS steady
workload, 3,513-RPS recovery burst, mixed background routes, and public 1Panel canary were not
completed before release. This limitation is explicitly disclosed rather than represented as
passing evidence. Large or latency-sensitive installations should canary v1.1.0 with their own
traffic and keep the v1.0.1 rollback reference available until their capacity and proxy behavior
are confirmed.

## Upgrade from v1.0.1

1. Back up MariaDB, persistent application storage, and the matching application key.
2. Review `TRUSTED_PROXIES` before recreating the container. Existing exact IP/CIDR values remain
   unchanged. An unset value in the bundled Compose files now becomes `*`; replace it with the
   proxy's exact observed IP/CIDR when direct backend access is possible.
3. Pull the `1.1.0` image and recreate only the API service.
4. Confirm `/api/version` reports `1.1.0`, `/up` succeeds, the admin login loads its CSS/JS over
   HTTPS, and application logs identify Nginx/PHP-FPM startup without configuration errors.
5. Run the public HTTPS checker and a representative RustDesk heartbeat/login test through the
   real reverse proxy before broad rollout.

```bash
docker compose pull rustdesk-api
docker compose up -d --force-recreate rustdesk-api
scripts/check-https-proxy.sh https://api.example.com
```

The bundled Compose files use a 30-second internal drain with a 35-second stop grace. Existing
custom Compose files keep the compatible eight-second internal default unless they opt into the
longer values. A single-replica recreation can still briefly interrupt new requests; deployments
that require uninterrupted long uploads or downloads need a proxy-drained multi-replica rollout.

## Rollback

No schema migration or storage-layout change prevents rollback. Pin the previous release and
recreate only the API service while retaining the same MariaDB, storage, and application key:

```bash
RUSTDESK_API_IMAGE='ghcr.io/gigaionllc/rustdesk-api-server:1.0.1@sha256:65fdd380ab101ef8fcf40e8281aa303257559f3da4008dfb00782138e71268e2' \
  docker compose up -d --force-recreate rustdesk-api
```

## Verification

The release candidate passed:

- 540 PHPUnit tests with 3,054 assertions against isolated MariaDB/InnoDB.
- Pint across 278 files and PHPStan across 178 files.
- 69 Playwright tests with 15 intentional project/screenshot skips.
- ESLint, checked-in vendor integrity, Composer/npm audits, Compose rendering, Bash/ShellCheck,
  workflow lint, immutable action-pin checks, and documentation-link validation.
- Native AMD64 and ARM64 builds, disposable-MariaDB runtime smoke, and final manifest/provenance
  checks in the exact-commit publication graph.
- A same-source 300-heartbeat-RPS Apache/Nginx comparison with zero failures, drops, or wire
  mismatches. The larger fleet-capacity qualification remains described in the capacity section.
