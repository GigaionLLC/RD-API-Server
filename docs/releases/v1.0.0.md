# RD-API-Server v1.0.0

**Release date:** 2026-07-17

**Status:** First stable release

RD-API-Server v1.0.0 is the first stable release of the independent, self-hosted API server and
administration console for compatible RustDesk clients. It combines the client-facing API,
remote-management features, and a modern web console in one PHP/Laravel application.

> RD-API-Server is an independent, third-party open-source project. It is not affiliated with,
> endorsed by, or sponsored by RustDesk or Purslane Ltd. RustDesk is referenced only to describe
> compatibility with the open-source client's public API.

## Release highlights

### Client API and remote management

- Account login with TOTP and email verification, plus OIDC/OAuth and LDAP/AD integrations.
- Device heartbeat, system-information reporting, deployment tokens, approval queues, and preset
  auto-registration.
- Strategy/settings synchronization, device and user-group access control, and live
  force-disconnect support.
- Personal and shared address books, granular peer and tag operations, device groups, and
  end-of-connection notes.
- Connection, file-transfer, login, alarm, and console-operation auditing, including session
  recording upload and playback controls.
- Scoped administration API keys, an OpenAPI 3.1 specification, and Postman and Bruno
  collections.

### Administration console

- Responsive dark and light interfaces built with Blade, jQuery, and Bootstrap 5; no SPA runtime
  is required.
- Dashboard telemetry, searchable device inventory, bulk assignment, live sessions, recordings,
  alarms, strategies, address books, users, groups, roles, webhooks, integrations, and settings.
- Delegated administration roles, destructive-action confirmation, accessible interaction
  states, and mobile navigation.

### Deployment and security

- Published multi-architecture Docker images for AMD64 and ARM64.
- MariaDB with InnoDB is the only supported database across runtime, development, tests, and CI.
- No default production administrator password; first-run production seeding rejects missing,
  short, known, placeholder, repeated, or username-derived passwords.
- Explicit reverse-proxy trust limited to sanitized `X-Forwarded-For` and
  `X-Forwarded-Proto` input from configured peers.
- Secure-cookie configuration, HTTPS proxy diagnostics, a fail-closed public proxy check, login
  and two-factor throttling, scoped authorization, and hardened credential storage.

## Install v1.0.0

Use the versioned release tag rather than the mutable `latest` tag when a deployment must remain
on this version:

```yaml
services:
  rustdesk-api:
    image: ghcr.io/gigaionllc/rustdesk-api-server:1.0.0
```

The release workflow also publishes the compatible `1.0` and `1` image tags. For the strongest
deployment reproducibility, pin the selected tag together with its published manifest digest.

Follow the [Quick Start](../../QUICKSTART.md) for the complete Docker and MariaDB configuration.
Production deployments require unique administrator and database passwords; do not copy example
placeholders into a live environment.

## HTTPS reverse-proxy requirement

TLS may terminate at 1Panel/OpenResty, Nginx Proxy Manager, Nginx, or another reverse proxy. The
private proxy-to-application hop may use HTTP, but the application must receive a sanitized
forwarded HTTPS scheme from an explicitly trusted peer.

Configure the public origin and trust boundary together:

```env
APP_URL=https://api.example.com
RUSTDESK_API_SERVER=https://api.example.com
TRUSTED_PROXIES=<proxy-address-seen-by-the-application>
SESSION_SECURE_COOKIE=true
```

Determine `TRUSTED_PROXIES` from the application access log for a request made through the public
proxy. Use the proxy's exact application-observed address whenever possible. If that address is
dynamic, trust only the narrow CIDR of a dedicated network shared exclusively by the application
and its trusted proxy.

Do not use `*`, `/0`, a broad private network, or an address shared with untrusted callers. The
backend port must be protected with an internal Docker network, a loopback binding, or a firewall
rule that permits only the proxy. The edge proxy must overwrite `Host` and
`X-Forwarded-Proto` and construct a trustworthy `X-Forwarded-For` chain.

After changing the environment, recreate the application container so Laravel rebuilds its
configuration cache, then verify the complete public path:

```bash
docker compose up -d --force-recreate <api-service>
scripts/check-https-proxy.sh https://api.example.com
```

The check validates the HTTPS login redirect, stylesheet and script URLs, secure login cookies,
and public stylesheet reachability. Mixed-content errors or an HTTPS request that redirects to an
HTTP URL indicate that the proxy address or forwarded-header configuration is still incorrect.

## Database and upgrade boundary

MariaDB with InnoDB is required. SQLite, Oracle MySQL, PostgreSQL, and SQL Server are not supported
runtime targets in v1.0.0.

Operators upgrading an installation that used the retired SQLite path must complete and validate
the [manual SQLite-to-MariaDB migration](../sqlite-to-mariadb.md) while running the last compatible
pre-v1 build. Do not start v1.0.0 against an unconverted SQLite database. The project deliberately
does not attempt an automatic best-effort conversion.

Before upgrading any existing installation:

1. Back up the MariaDB database, application storage, and the matching application key.
2. Confirm every application table uses InnoDB.
3. Replace a legacy `DB_CONNECTION=mysql` value with `DB_CONNECTION=mariadb` only after the
   InnoDB audit passes.
4. Expand any credential-bearing `DB_URL` into the discrete database settings, then remove
   `DB_URL`.
5. Preserve the existing `APP_KEY`; the database and application key form one recovery unit.

See the [Quick Start database section](../../QUICKSTART.md#4-database-support-and-upgrades) for
the complete preflight and rollback procedure.
