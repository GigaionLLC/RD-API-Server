## 1. Intent and scope

Ship a backward-compatible patch release that makes dark mode the first-visit default while
preserving an explicit light-mode choice, and removes the 35-minute ARM emulation cliff from cold
multi-architecture container builds without weakening image, dependency, or workflow security.

In scope:

- Dark by default on login, two-factor challenge, and authenticated administration pages.
- Persisted `rd_theme=light` and `rd_theme=dark` choices continue to override the default.
- Native AMD64 and ARM64 release builds, one PHP extension compilation per architecture, and
  dependency layers that survive application-only changes.
- SHA-pinned actions, digest-pinned build inputs, locked Composer/npm dependencies, provenance,
  per-architecture runtime checks, and CI-gated public image promotion.

Out of scope:

- New theme colors or layout changes.
- Database schema, RustDesk wire-contract, or application feature changes.
- Rewriting or moving the published `v1.0.0` tag.

## 2. Evidence and constraints

- The three Blade entry points and `public/assets/js/app.js` currently use the OS color preference
  when no theme is stored, then follow live OS preference changes.
- Explicit stored light/dark choices already work and must remain stable.
- The cold Docker Publish run compiled nearly identical PHP extension sets twice per architecture.
  ARM64 QEMU compilation consumed roughly 35 minutes; a warm cache reduced the same publish to
  about 82 seconds.
- `docker/Dockerfile.runtime` copies the full source tree before Composer installation, invalidating
  the vendor layer for ordinary source or documentation changes.
- GHA BuildKit caching is enabled but can be evicted; architecture-specific cache scopes and a
  durable registry fallback are required.

## 3. Execution plan

### Theme behavior

- Replace the no-stored-preference branch with literal dark mode in:
  - `resources/views/layouts/admin.blade.php`
  - `resources/views/admin/login.blade.php`
  - `resources/views/admin/two_factor/challenge.blade.php`
  - `public/assets/js/app.js`
- Remove automatic OS-theme following while leaving storage parsing, toggling, and persistence
  intact.
- Make the Playwright `desktop-light` project seed an explicit saved light preference so light-mode
  accessibility and interaction coverage remains real after the default changes.
- Add browser tests for dark default under an OS-light browser, login/dashboard consistency,
  saved-light persistence, reload persistence, and immunity to later OS preference changes.
- Update `Wiki/core/06-design-system.md` with the theme contract.

### Container build and publish

- Refactor `docker/Dockerfile.runtime` around one digest-pinned PHP-Apache extension stage used by
  both Composer dependency assembly and the final runtime. Do not copy compiled modules between
  different PHP base variants.
- Copy `composer.json` and `composer.lock` before application source, install locked dependencies
  without scripts/autoload, then copy source and generate the optimized script-free autoloader.
- Build AMD64 on `ubuntu-24.04` and ARM64 on `ubuntu-24.04-arm`; remove QEMU.
- Push architecture results by digest, smoke-test each native image, and merge only successful
  digests into final `latest`, SemVer, and SHA tags with provenance retained.
- Use architecture-specific GHA cache scopes plus GHCR registry-cache fallbacks. Add concurrency
  that cancels superseded main builds without cancelling release tags.
- Prevent documentation-only main pushes from rebuilding the runtime image, while release tags
  always publish.
- Ensure public tag promotion depends on successful quality gates for the exact commit.
- Document cache maintenance, native-runner behavior, cold/warm expectations, and digest checks in
  `docker/README.md`.
